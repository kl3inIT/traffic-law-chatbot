<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Admin API -> Source Service | Request inputs must be validated before mutating source state |
| Source lifecycle state machine | State transitions must be enforced in service, not just by convention |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-02-01 | Tampering | SourceService state transitions | mitigate | Explicit precondition checks in approve/activate: throw AppException if source is not in expected state. Tested in SourceServiceTest (tests 4, 7). |
| T-02-02 | Repudiation | Source approval/rejection | mitigate | Every state change persists a KbSourceApprovalEvent record with action, previousState, newState, actedBy, and actedAt. Tests 3 and 5 verify event creation. |
| T-02-03 | Information Disclosure | SourceDetailResponse | accept | Phase 1 has no auth layer; admin APIs are internal only. Auth added in a later phase (see ROADMAP Phase 4 PLAT-02). |
| T-02-04 | Elevation of Privilege | SourceAdminController | accept | No end-user roles in Phase 1. All /api/v1/admin/* endpoints are admin-only by convention. Auth middleware is a Phase 4 concern. |
| T-02-05 | Spoofing | createdBy / actedBy fields | accept | These fields are free-text in Phase 1; no identity verification. Acceptable for backend-foundation phase. Real identity binding comes with auth in Phase 4. |
</threat_model>
