## Spacing Scale

Standard 8-point scale applied across all admin screens. Source: established pattern in sources/parameters/trust-policy pages.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge inner padding, inline element gaps |
| sm | 8px | Compact element spacing, button gaps within toolbars |
| md | 16px | Default element spacing, form field gaps, card inner padding |
| lg | 24px | Section padding (page container: `p-6`) |
| xl | 32px | Layout gaps between major sections |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing (not used in admin screens) |

Exceptions:
- Left-panel list width: fixed `w-64` (256px) — matches parameters and trust-policy master-detail panels
- DataTable skeleton rows: `h-12` per row, `space-y-2` gap
- Filter toolbar: `px-4 py-2` — matches sources page bulk-action toolbar

## Typography

Inherited from established admin screens. Do not redefine — extend these roles only.

Exactly 2 font weights are used: **400 (normal)** and **600 (semibold)**.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 14px (text-sm) | 400 (normal) | 1.5 | Table cell content, form labels, descriptions |
| Label | 12px (text-xs) | 600 (semibold) | 1.4 | Form field labels (`<Label>`), filter labels — distinguished from Body by smaller size and semibold weight |
| Heading | 20px (text-xl) | 600 (semibold) | 1.2 | Page titles (`h1 text-xl font-semibold`) |
| Caption | 12px (text-xs) | 400 (normal) | 1.4 | Pagination counters, muted metadata, token counts — distinguished from Label by normal weight and `text-muted-foreground` color |

Font stack:
- UI text: `var(--font-sans)` (Geist Sans)
- YAML/code content: `font-mono text-sm` (Geist Mono) — already used in parameters YAML textarea

### Screen 1: Chat Log List — `(admin)/chat-logs/page.tsx`

**Layout:** Full-width DataTable page. Pattern: sources page (`space-y-4 p-6`).

**Toolbar (above table):**
- Left: page heading `h1 text-xl font-semibold` — "Lịch sử hội thoại"
- Right: no primary CTA (logs are append-only, no create action)

**Filter bar (below heading, above table):**
- Date range: two `<Input type="date">` fields labeled "Từ ngày" / "Đến ngày", inline with `<Label>`
- Grounding status: `<Select>` dropdown — options: "Tất cả", "GROUNDED", "LIMITED_GROUNDING", "REFUSED"
- Question text search: `<Input type="text">` with placeholder "Tìm kiếm theo câu hỏi..."
- Filter bar container: `bg-muted flex items-center gap-3 rounded-md border px-4 py-2` — matches sources bulk-action bar

**DataTable columns:**
| Column | Width | Content |
|--------|-------|---------|
| Ngày tạo | 160px | `createdDate` formatted as `dd/MM/yyyy HH:mm` |
| Câu hỏi | auto (flex) | `question` text, truncated with `truncate max-w-xs` |
| Trạng thái | 140px | `groundingStatus` as `<Badge variant="outline">` with semantic color |
| Token (P+C) | 120px | `promptTokens + completionTokens` formatted as `"P: {n} / C: {n}"` |
| Thời gian (ms) | 120px | `responseTime` in milliseconds |

**Row interaction:** Click anywhere on row navigates to `(admin)/chat-logs/[id]/page.tsx`.

**Loading state:** `<QueryBoundary>` wrapping DataTable, skeleton `h-12` rows x8.

**Empty state:**
- Heading: "Chưa có lịch sử hội thoại"
- Body: "Các cuộc hội thoại sẽ xuất hiện ở đây khi người dùng bắt đầu trò chuyện."

**Pagination:** Same pattern as sources page — "Trang {n} / {total} ({count} bản ghi)" with Trước/Sau buttons.

## Interaction Patterns

### Filter behavior (Chat Log List)
- Date range and grounding status filters are applied server-side via query params on the TanStack Query key.
- Text search input debounced 300ms before triggering refetch.
- Filters are independent — any combination is valid.
- Filter state lives in component `useState`, reset on page unmount.
- No "Apply" button needed — filters trigger automatically on change (except text search which uses debounce).

### Run checks async flow
1. Admin clicks "Chạy kiểm tra" button
2. Button enters loading state (spinner + disabled)
3. POST to `/api/v1/admin/checks/runs` returns a run ID immediately
4. Button returns to normal
5. Navigate to `(admin)/checks/runs/` list — the new run appears with status RUNNING
6. Admin manually refreshes or the list auto-refetches on a 5-second interval while any run has RUNNING status
7. No toast or notification system required for v1

### Check run list auto-poll
- `useQuery` with `refetchInterval: 5000` when any run in the list has `status === 'RUNNING'`
- Stops polling when all runs are in a terminal state (COMPLETED / FAILED)

### Check definition active toggle
- `<Checkbox>` with `<Label>` inline — `flex items-center gap-2`
- Unchecking active means this definition is excluded from future runs
- No confirmation required for the toggle — it is non-destructive

### Delete confirmation (CheckDef)
- `<AlertDialog>` opened by clicking "Xóa" button in form footer
- Destructive `<AlertDialogAction>` button uses `bg-destructive text-destructive-foreground` — matches existing pattern
