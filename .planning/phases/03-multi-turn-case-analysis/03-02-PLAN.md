<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user turn -> fact extraction | Untrusted natural-language input is transformed into durable explicit facts. |
| thread state -> clarification policy | Active facts and pending facts determine whether legal conclusions are allowed. |
| thread-aware orchestration -> retrieval | Follow-up retrieval must stay grounded in approved/trusted/active sources even when memory exists. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-02-01 | T | FactMemoryService | mitigate | Persist facts only from user-role messages and mark each stored fact with source message provenance so assistant-generated text cannot become memory. |
| T-03-02-02 | T | correction handling | mitigate | Implement `ACTIVE`/`SUPERSEDED` status transitions per fact key so the latest corrected fact wins without erasing audit history. |
| T-03-02-03 | I | clarification-needed response | mitigate | Return `pendingFacts` plus no final outcome/conclusion fields in blocking mode so the API does not overstate unsupported legal results. |
| T-03-02-04 | D | clarification loop | mitigate | Add config-backed clarification cap and tests for fallback to `REFUSED` after repeated unresolved turns. |
| T-03-02-05 | E | memory bypassing legal grounding | mitigate | Keep retrieval on every substantive turn and preserve `RetrievalPolicy.RETRIEVAL_FILTER`; remembered facts may shape the query but never replace legal sources. |
</threat_model>
