<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| public client -> thread endpoints | Untrusted text and thread IDs enter persistence and orchestration entrypoints. |
| controller/service -> database | Thread IDs, message rows, and fact rows cross into durable storage and must preserve auditability. |
| thread-aware response contract -> future frontend | Phase 4 will depend on stable mode/thread fields without losing current citation/disclaimer fields. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01-01 | S | thread continuation endpoint | mitigate | Use UUID-backed thread IDs, validate path params, and return app-level not-found errors for unknown IDs rather than exposing internal lookup behavior. |
| T-03-01-02 | T | message/fact persistence model | mitigate | Keep immutable `chat_message` history separate from mutable `thread_fact` rows with explicit `ACTIVE`/`SUPERSEDED` states per D-07/D-09. |
| T-03-01-03 | R | thread history audit trail | mitigate | Store every user/assistant turn as append-only messages with timestamps and source linkage so later corrections do not erase the prior record. |
| T-03-01-04 | I | response contract evolution | mitigate | Extend `ChatAnswerResponse` in place so citations, sources, disclaimer, and grounding fields remain visible while new thread metadata is added. |
| T-03-01-05 | D | malformed payload path | mitigate | Add MockMvc validation coverage for oversized/blank questions and invalid UUIDs so bad input fails fast via ProblemDetail instead of reaching persistence. |
| T-03-01-06 | E | parallel chat stack creep | mitigate | Implement all new endpoints under the existing `chat` module and `ApiPaths`, explicitly forbidding a separate scenario-only controller family. |
</threat_model>
