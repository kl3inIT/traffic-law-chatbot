## User Constraints (from CONTEXT.md)

### Locked Decisions
### Refactor scope
- **D-01:** Phase 01.1 uses a hybrid refactor strategy, not a full rewrite and not a full keep-custom approach.
- **D-02:** The custom ingestion domain boundary stays in place: `ParsedDocument`, async orchestration, source/source_version/job separation, approval/trusted/active retrieval gating, and provenance persistence remain first-class.
- **D-03:** The immediate implementation focus is to fix `final_url` tracking and refactor URL HTML ingestion first.
- **D-04:** PDF/file parser replacement can be evaluated in this phase only as an incremental follow-on behind the same custom boundary; chunking changes are lower priority and should come last.

### Provenance and metadata contract
- **D-05:** Metadata preservation is strict. The refactor must preserve retrieval-compatible metadata behavior, including source linkage, page/section or URL location hints, processing version, parser identity, and approval/trust compatibility.
- **D-06:** Chunk-level audit fields such as ordinal and content hash remain required unless a replacement path can preserve them equivalently without breaking current inspection/admin flows.
- **D-07:** URL ingestion must correctly persist both requested URL and final resolved URL as provenance data.

### URL safety and parsing boundary
- **D-08:** The network trust boundary remains custom. SSRF validation, host checks, fetch provenance, and final URL capture must stay outside Spring AI readers.
- **D-09:** Spring AI HTML parsing may be introduced only after a custom safe fetch step has validated and fetched the URL content.
- **D-10:** URL ingestion remains page-by-page import only; no crawler or broader website discovery behavior is added in this phase.

### Migration shape
- **D-11:** Rollout should be bugfix-first and adapter-based. Fix the `final_url` bug first, then introduce hybrid adapters behind the existing parser abstractions.
- **D-12:** Existing API contracts, job visibility, and retrieval policy behavior should remain stable during migration.
- **D-13:** Avoid broad one-phase replacement of URL parser, file parsers, and chunker together unless research shows the risk is low and metadata behavior is preserved.

### Spring AI adoption direction
- **D-14:** Use Spring AI for commodity ETL internals where it reduces maintenance without taking ownership of domain policy.
- **D-15:** Preferred evaluation order is: safe custom URL fetch + Spring AI HTML reader first, then page-aware PDF reader if it preserves page provenance adequately, then token-based splitting only if metadata carry-through remains clean.

- Whether PDF evaluation lands in the same plan set or a follow-on plan within this phase.
- Exact split between parser adapter responsibilities and orchestrator responsibilities, as long as the locked boundary and metadata guarantees are preserved.

### Deferred Ideas (OUT OF SCOPE)
- Broad website crawling or seed-based discovery remains out of scope.
- Full parser/chunker stack replacement in one pass is deferred unless research proves metadata and rollout risk are acceptable.
- Chat-facing changes and any retrieval behavior expansion remain outside this phase.

## Summary

The current code already has the right macro-boundary for a low-risk hybrid migration: `IngestionService` validates and queues work, `IngestionOrchestrator` owns async step transitions and vector-store writes, `ParsedDocument` is the parser output contract, and `TextChunker` produces the audit metadata that downstream inspection relies on. The main defect is that URL provenance is collected incorrectly: `UrlPageParser` computes `finalUrl`, but `ParsedDocument` cannot carry it, and `IngestionOrchestrator` persists `final_url` as `version.getCanonicalUrl()` regardless of redirects. [VERIFIED: codebase D:\ai\traffic-law-chatbot\src\main\java\com\vn\traffic\chatbot\ingestion\parser\UrlPageParser.java] [VERIFIED: codebase D:\ai\traffic-law-chatbot\src\main\java\com\vn\traffic\chatbot\ingestion\orchestrator\IngestionOrchestrator.java] [VERIFIED: codebase D:\ai\traffic-law-chatbot\src\main\java\com\vn\traffic\chatbot\ingestion\parser\ParsedDocument.java]

The lowest-risk migration is therefore not to let Spring AI own fetching. Instead, keep a custom safe-fetch stage that performs scheme checks, SSRF host validation, redirect-aware provenance capture, and response snapshot persistence, then hand the fetched HTML bytes/string to a Spring AI HTML reader adapter that returns a `ParsedDocument`. Spring AI’s ETL model is interface-based, `JsoupDocumentReader` is officially documented for HTML, and `TokenTextSplitter` preserves source metadata when splitting, but this codebase still needs custom adaptation because its metadata contract includes parser identity, page/section hints, chunk ordinal, content hash, and retrieval/filter compatibility beyond Spring AI’s default abstractions. [VERIFIED: codebase D:\ai\traffic-law-chatbot\src\main\java\com\vn\traffic\chatbot\ingestion\service\IngestionService.java] [CITED: https://docs.spring.io/spring-ai/reference/api/etl-pipeline.html] [CITED: https://docs.spring.io/spring-ai/reference/api/vectordbs.html]

PDF evaluation should be included only as a bounded follow-on plan inside Phase 01.1, not mixed into the initial URL refactor plan. Spring AI documents `PagePdfDocumentReader` and `ParagraphPdfDocumentReader`, which makes evaluation worthwhile, but current shipped behavior depends on `TikaDocumentParser` plus custom page splitting on form-feed, so replacing PDF parsing in the same first cut would combine two provenance-sensitive migrations at once. [VERIFIED: codebase D:\ai\traffic-law-chatbot\src\main\java\com\vn\traffic\chatbot\ingestion\parser\TikaDocumentParser.java] [CITED: https://docs.spring.io/spring-ai/reference/api/etl-pipeline.html]

**Primary recommendation:** Fix URL provenance first by introducing a custom `FetchedUrlResource`/`FetchResult` boundary, then add a Spring AI HTML-reader adapter behind `ParsedDocument`, and defer PDF replacement to a second plan in the same phase. [VERIFIED: codebase D:\ai\traffic-law-chatbot\src\main\java\com\vn\traffic\chatbot\ingestion\orchestrator\IngestionOrchestrator.java] [CITED: https://docs.spring.io/spring-ai/reference/api/etl-pipeline.html]

## Common Pitfalls

### Pitfall 1: Fixing `final_url` only at the snapshot write site
**What goes wrong:** The immediate bug disappears only superficially if `KbSourceFetchSnapshot.finalUrl` is patched but parser/output types still cannot carry resolved URL data through the flow. [VERIFIED: codebase D:\ai\traffic-law-chatbot\src\main\java\com\vn\traffic\chatbot\ingestion\orchestrator\IngestionOrchestrator.java] [VERIFIED: codebase D:\ai\traffic-law-chatbot\src\main\java\com\vn\traffic\chatbot\ingestion\parser\ParsedDocument.java]
**Why it happens:** `UrlPageParser` computes `finalUrl`, but the returned `ParsedDocument` drops it, and the orchestrator currently hardcodes the canonical URL into `finalUrl`. [VERIFIED: codebase D:\ai\traffic-law-chatbot\src\main\java\com\vn\traffic\chatbot\ingestion\parser\UrlPageParser.java] [VERIFIED: codebase D:\ai\traffic-law-chatbot\src\main\java\com\vn\traffic\chatbot\ingestion\orchestrator\IngestionOrchestrator.java]
**How to avoid:** Introduce a dedicated fetch-result object that explicitly carries `requestedUrl`, `resolvedUrl`, `httpStatus`, headers of interest, and response content. [CITED: https://jsoup.org/apidocs/org/jsoup/Connection.Base.html] [ASSUMED]
**Warning signs:** Redirected imports still persist identical requested/final URLs in tests or admin inspection. [ASSUMED]

### 1. Lowest-risk hybrid migration path for URL HTML ingestion
1. Introduce a custom fetch component separate from parsing, for example `SafeUrlFetcher`, that returns a fetch result object containing request URL, resolved URL, status/headers, content type, and body bytes/string. [ASSUMED]
2. Move Jsoup network access out of `UrlPageParser` into that fetch component so SSRF checks and provenance capture remain outside Spring AI. [VERIFIED: phase context] [VERIFIED: codebase D:\ai\traffic-law-chatbot\src\main\java\com\vn\traffic\chatbot\ingestion\parser\UrlPageParser.java]
3. Replace only the HTML text-extraction internals with a Spring AI adapter that consumes the already-fetched HTML resource and maps the resulting Spring AI `Document` back into `ParsedDocument`. [CITED: https://docs.spring.io/spring-ai/reference/api/etl-pipeline.html] [ASSUMED]

### 2. How `final_url` tracking should be fixed
