# Phase 01.1 Plan 03: Run bounded PDF parser parity evaluation and adopt Spring AI PDF only if provenance compatibility is proven Summary

Spring AI PDF parsing now runs behind a guarded adapter for PDF uploads only, with automated parity checks proving page-aware provenance, parser audit fields, and downstream chunk metadata compatibility while leaving chunking and non-PDF APIs unchanged.

## What Changed

- Added `SpringAiPdfParser` as a bounded Spring AI adapter using `PagePdfDocumentReader` and mapping each page into `ParsedDocument.PageSection` with `page-{n}` section refs.
- Updated `IngestionOrchestrator` to:
  - use `SafeUrlFetcher` plus `UrlPageParser.parseFetchedPage()` for the already-planned URL HTML path,
  - select `SpringAiPdfParser` only for PDF file inputs,
  - fall back to `TikaDocumentParser` for non-PDF or unsupported uploads,
  - persist parser name/version for both URL and file ingestion paths.
- Added PDF parity and parser selection tests covering:
  - page-aware Spring AI PDF section mapping,
  - explicit parser provenance fields,
  - unchanged chunk metadata keys through `TextChunker`,
  - deterministic fallback to Tika for non-PDF paths.
- Added the explicit `spring-ai-pdf-document-reader` dependency required by the plan.

## Decisions Made

1. `PagePdfDocumentReader` was chosen over broader PDF migration because it can preserve page identity cleanly through `ParsedDocument.PageSection`.
2. Parser selection remains bounded to file ingestion and only switches for PDFs, preserving the current Tika path for Word and other file types.
3. Chunking stayed unchanged; compatibility is enforced in tests via existing `chunkOrdinal`, `pageNumber`, `sectionRef`, and `contentHash` expectations.
