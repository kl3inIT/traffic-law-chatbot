# Phase 01.1 Plan 04: Promote Spring AI ingestion path and token chunking Summary

Phase 01.1 is now complete. Spring AI readers are the primary ingestion path for covered formats, parser selection is isolated in a resolver, chunking now uses the Spring AI token splitter, and chunking provenance is persisted without breaking retrieval or inspection metadata compatibility.

## What Changed

- Renamed implementation-branded ingestion components to domain-oriented names:
  - `SpringAiHtmlParser` -> `HtmlDocumentParser`
  - `SpringAiPdfParser` -> `PdfDocumentParser`
  - `TextChunker` -> `TokenChunkingService`
- Extracted file parser selection from `IngestionOrchestrator` into `FileIngestionParserResolver`.
- Made Spring AI the default ingestion path for covered formats:
  - HTML remains behind `UrlPageParser` and safe-fetch boundaries.
  - PDF uses `PdfDocumentParser` as the primary path.
  - Tika is constrained to explicit fallback usage for DOC/DOCX and unsupported parity gaps.
- Moved chunking to Spring AI token-based splitting via `TokenTextSplitter.builder()` while preserving:
  - `chunkOrdinal`
  - `pageNumber`
  - `sectionRef`
  - `contentHash`
  - `processingVersion`
  - `sourceId`
  - `sourceVersionId`
- Persisted chunking provenance on `KbSourceVersion` via:
  - `chunkingStrategy = spring-ai-token-text-splitter`
  - `chunkingVersion = 2.0.0-M4`
- Removed the Gradle Groovy DSL repository warning by switching repository URL syntax to assignment form.

## Decisions Made

1. Keep Spring AI inside repo-owned boundaries by translating reader output into `ParsedDocument` and preserving orchestrator-owned metadata/indexing logic.
2. Constrain Tika to explicit fallback handling instead of leaving it as the default file ingestion path.
3. Persist chunking identity on `KbSourceVersion` so admins can audit both parser and chunking provenance.
4. Use the non-deprecated `TokenTextSplitter.builder()` API instead of deprecated constructors.

## Deviations from Plan

None requiring rollback or follow-up. The only implementation adjustment was switching from deprecated `TokenTextSplitter` constructors to the builder API after confirming the available Spring AI 2.0.0-M4 class signatures on the local classpath.
