# Phase 06.1: multi-provider-ai-model-selection - Context

**Gathered:** 2026-04-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Integrate 9router as the sole AI gateway (OpenAI-compatible, localhost:20128), replace the hardcoded `AllowedModel` enum with a YAML-driven model catalog, build a `Map<String, ChatClient>` at startup, wire `modelId` from chat request body into `ChatService`, wire `evaluatorModel` from active `AiParameterSet` into `LlmSemanticEvaluator`, expose `GET /api/admin/allowed-models`, and add model selection dropdowns to the chat UI header and admin Parameters page.

Out of scope: live model swap without restart, per-message model switching, multi-model parallel queries.

</domain>

<decisions>
## Implementation Decisions

### 9router Configuration
- **D-01:** `spring.ai.openai.base-url=${OPENAI_BASE_URL:http://localhost:20128}` — 9router runs locally on port 20128, overridable via env var.
- **D-02:** `spring.ai.openai.api-key=${OPENAI_API_KEY:none}` — 9router accepts any string; keep the existing env var.
- **D-03:** Model catalog defined in YAML following the `pacphi/spring-ai-openrouter-example` pattern — **not an enum**:
  ```yaml
  app:
    ai:
      chat-model: ${CHAT_MODEL:claude-sonnet-4-6}
      evaluator-model: ${EVALUATOR_MODEL:claude-haiku-4-5-20251001}
      models:
        - id: gpt-5.4
          display-name: "GPT-5.4"
        - id: claude-sonnet-4-6
          display-name: "Claude Sonnet 4.6"
        - id: claude-haiku-4-5-20251001
          display-name: "Claude Haiku 4.5"
  ```
- **D-04:** `AllowedModel` enum (`parameter/domain/AllowedModel.java`) **deleted entirely**, replaced by an `AiModelProperties` `@ConfigurationProperties` record reading `app.ai.*`.
- **D-05:** Adding a model requires only a YAML edit and restart — no recompile needed.

### ChatClient Factory
- **D-06:** `ChatClientConfig` builds `Map<String, ChatClient>` at startup from `app.ai.models` — one entry per model ID. Each `OpenAiChatModel` is constructed with `OpenAiChatOptions.builder().model(modelId).build()`.
- **D-07:** `ChatService` receives an optional `modelId` from the request body. Looks up the Map; if null or not found → falls back to the `app.ai.chat-model` config value.
- **D-08:** `LlmClarificationService` uses the same chat model (Map lookup with `app.ai.chat-model` as default) — consistent with the main chat flow.
- **D-09:** `LlmSemanticEvaluator` reads `evaluatorModel` from the active `AiParameterSet` (via `AiParameterSetService.getActive()`). If no active param set or `evaluatorModel` is null → falls back to `app.ai.evaluator-model` config.
- **D-10:** No per-request DB read for chat model — model comes from the request body. Only the evaluator reads DB (active param set).

### Fallback & Default Model
- **D-11:** Chat model fallback chain: `request.modelId` → `app.ai.chat-model` config. Never throws an error due to missing modelId.
- **D-12:** Unrecognized `modelId` (not in catalog) → log warning + fallback to default. Do not return HTTP 400.
- **D-13:** Evaluator model fallback chain: `activeParamSet.evaluatorModel` → `app.ai.evaluator-model` config.

### API & Frontend
- **D-14:** `GET /api/admin/allowed-models` — new endpoint in `AiParameterSetController` (path constant `ApiPaths.ALLOWED_MODELS` already declared at line 66). Returns `List<{id, displayName}>` from `AiModelProperties.models()`. **Unblocks Phase 5 UAT skip (test 9).**
- **D-15:** `ChatQuestionRequest` gains a nullable `String modelId` field (not required, no validation).
- **D-16:** Chat UI: model dropdown in the **chat page header**. Fetches list from `/api/admin/allowed-models`. Selected value sent in each request body.
- **D-17:** Admin Parameters page: `chatModel` and `evaluatorModel` dropdowns fetch from the same `/api/admin/allowed-models`. Replace any existing text inputs with a Select component.

### Claude's Discretion
- How to inject `Map<String, ChatClient>` into services (bean name, qualifier, or typed wrapper).
- Caching strategy for `AiParameterSetService.getActive()` if needed (e.g., short-lived `@Cacheable`).
- Class name and package location for the `@ConfigurationProperties` bean.

</decisions>

<specifics>
## Specific Ideas

- Follow **exactly the pattern** from `pacphi/spring-ai-openrouter-example`: model list in YAML, `Map<String, ChatClient>` built at startup, base-url override via env var.
- 9router endpoint: `http://localhost:20128` (default), override via `OPENAI_BASE_URL` env var.
- Three models currently available via 9router: `gpt-5.4`, `claude-sonnet-4-6`, `claude-haiku-4-5-20251001`.
- Model changes are infrequent → restart on change is acceptable → no live reload needed.
- Chat model dropdown sits in the **chat page header**, not in the chat input area.
- Both chat UI and admin Parameters page use `/api/admin/allowed-models` to fetch the model list.

</specifics>

<deferred>
## Deferred Ideas

- Live model swap without restart (dynamic DB read per request) — can be added later if needed.
- Per-message model switching within a conversation — out of v1 scope.
- Multi-model parallel query (like MultiChat in pacphi repo) — out of scope.
- User auth/authorization for model selection — out of v1 scope.

</deferred>
