## Summary

This phase replaces the hardcoded `AllowedModel` enum with a YAML-driven model catalog and builds a `Map<String, ChatClient>` factory that routes requests to 9router (OpenAI-compatible gateway on localhost:20128). The Spring AI `spring-ai-starter-model-openai` dependency already present in `build.gradle` supports base-url override via `spring.ai.openai.base-url`, which is the core mechanism that redirects all model calls to 9router. Each model entry in `app.ai.models` gets its own `OpenAiChatModel` instance with `OpenAiChatOptions` specifying the model ID — all sharing the same 9router base URL and API key.

The backend changes are self-contained: new `@ConfigurationProperties` record, rebuilt `ChatClientConfig`, and minor additions to three services. The frontend changes add a model dropdown to the chat header (shared state across chat pages) and two Select dropdowns in the Parameters page form.

**Primary recommendation:** Follow the `pacphi/spring-ai-openrouter-example` pattern exactly — build the Map at startup using a `@Configuration` class, inject the Map by type or with a `@Qualifier`, and use the default model ID as the fallback key in every lookup.

## Architecture Patterns

### Pattern 1: YAML-Driven Model Catalog with `@ConfigurationProperties`

**What:** A record bound to `app.ai.*` that holds `chatModel`, `evaluatorModel`, and the `List<ModelEntry>` catalog.

**When to use:** Startup-time binding of structured YAML config — no runtime DB needed.

**Example:**
```java
// Source: Phase 4.1 AppProperties pattern + Spring Boot @ConfigurationProperties docs
@ConfigurationProperties(prefix = "app.ai")
public record AiModelProperties(
    String chatModel,
    String evaluatorModel,
    List<ModelEntry> models
) {
    public record ModelEntry(String id, String displayName) {}
}
```

Enable with `@EnableConfigurationProperties(AiModelProperties.class)` in the main application class or in a `@Configuration` class. [ASSUMED — standard Spring Boot pattern, matches existing AppProperties usage]

### Pattern 2: Map<String, ChatClient> Built at Startup

**What:** A `@Bean` of type `Map<String, ChatClient>` in `ChatClientConfig`, keyed by model ID. Each entry wraps a dedicated `OpenAiChatModel` configured with that model's ID via `OpenAiChatOptions`.

**Key Spring AI classes needed:**
- `OpenAiApi` — the HTTP client, shared across all models (same base-url + api-key)
- `OpenAiChatModel` — wraps `OpenAiApi` with per-model `OpenAiChatOptions`
- `ChatClient.builder(chatModel).build()` — wraps `OpenAiChatModel` into a `ChatClient`

**Example:**
```java
// Source: pacphi/spring-ai-openrouter-example pattern + Spring AI OpenAI autoconfigure
@Configuration
@EnableConfigurationProperties(AiModelProperties.class)
public class ChatClientConfig {

    @Bean
    public Map<String, ChatClient> chatClientMap(
            OpenAiApi openAiApi,
            AiModelProperties modelProperties,
            ChatClient.Builder chatClientBuilder
    ) {
        Map<String, ChatClient> map = new LinkedHashMap<>();
        for (AiModelProperties.ModelEntry entry : modelProperties.models()) {
            OpenAiChatOptions options = OpenAiChatOptions.builder()
                    .model(entry.id())
                    .build();
            OpenAiChatModel chatModel = new OpenAiChatModel(openAiApi, options);
            map.put(entry.id(), ChatClient.builder(chatModel).build());
        }
        return map;
    }
}
```

**Note:** `OpenAiApi` is auto-configured by `spring-ai-starter-model-openai` using `spring.ai.openai.base-url` and `spring.ai.openai.api-key`. Reusing the auto-configured bean means all models share the same 9router endpoint. [ASSUMED — based on Spring AI OpenAI autoconfiguration patterns; consistent with pacphi pattern]

### Pattern 3: Model Lookup with Fallback in Services

**What:** Services receive the `Map<String, ChatClient>` by injection, plus the `AiModelProperties` for default model IDs.

**Example for ChatService:**
```java
// ChatService — resolves modelId from request, falls back to config default
private ChatClient resolveClient(String requestedModelId) {
    String effectiveId = (requestedModelId != null && chatClientMap.containsKey(requestedModelId))
            ? requestedModelId
            : modelProperties.chatModel();
    ChatClient client = chatClientMap.get(effectiveId);
    if (client == null) {
        log.warn("No ChatClient found for modelId '{}', using any available client", effectiveId);
        return chatClientMap.values().iterator().next(); // last-resort fallback
    }
    return client;
}
```

**Example for LlmSemanticEvaluator:**
```java
// LlmSemanticEvaluator — reads evaluatorModel from active AiParameterSet
private ChatClient resolveEvaluatorClient() {
    String modelId = parameterSetService.getActive()
            .map(AiParameterSet::getEvaluatorModel)
            .filter(m -> m != null && !m.isBlank())
            .orElse(modelProperties.evaluatorModel());
    return chatClientMap.getOrDefault(modelId,
            chatClientMap.get(modelProperties.evaluatorModel()));
}
```

### Pattern 4: Injecting Map<String, ChatClient> into Services

**Claude's Discretion area — recommended approach:** Inject as `Map<String, ChatClient>` by type directly. Spring disambiguates generic types correctly. No `@Qualifier` needed if only one `Map<String, ChatClient>` bean exists.

```java
@Service
@RequiredArgsConstructor
public class ChatService {
    private final Map<String, ChatClient> chatClientMap;  // injected by type
    private final AiModelProperties aiModelProperties;
    // ...
}
```

**Alternative if ambiguity arises:** Use a typed wrapper bean:
```java
@Bean
@Qualifier("chatClientMap")
public Map<String, ChatClient> chatClientMap(...) { ... }
// Then inject with @Qualifier("chatClientMap")
```

### Pattern 5: GET /api/admin/allowed-models Endpoint

**What:** New method in `AiParameterSetController` (which already handles `/api/v1/admin/parameter-sets`). Use a separate `@GetMapping` on `ApiPaths.ALLOWED_MODELS` path. Since `ALLOWED_MODELS = "/api/v1/admin/allowed-models"` and the controller is mapped to `PARAMETER_SETS = "/api/v1/admin/parameter-sets"`, the new endpoint should either:

- Option A: Add a new `@RestController` at `ApiPaths.ALLOWED_MODELS` (clean separation)
- Option B: Add a `@GetMapping` on a different base path directly in `AiParameterSetController` using the full path

**Recommended (Claude's Discretion):** Create a minimal `AllowedModelsController` to avoid polluting `AiParameterSetController` with an unrelated concern. One `@GetMapping` method, no service delegation needed — reads directly from injected `AiModelProperties`.

```java
@RestController
@RequiredArgsConstructor
public class AllowedModelsController {
    private final AiModelProperties modelProperties;

    @GetMapping(ApiPaths.ALLOWED_MODELS)
    public ResponseEntity<List<AllowedModelResponse>> list() {
        List<AllowedModelResponse> models = modelProperties.models().stream()
                .map(m -> new AllowedModelResponse(m.id(), m.displayName()))
                .toList();
        return ResponseEntity.ok(models);
    }
}
```

**Response DTO:**
```java
public record AllowedModelResponse(String modelId, String displayName) {}
```

Note: `AllowedModel` interface is already defined in `frontend/types/api.ts` with fields `modelId` and `displayName` — backend DTO must match this contract. [VERIFIED: frontend/types/api.ts line 308-311]

### Pattern 6: Frontend Model Dropdown in Chat Header

**What:** A model selector rendered in the chat page header area. Since both `(chat)/page.tsx` and `(chat)/threads/[id]/page.tsx` need to share selected model state, the cleanest approach is React context or lifting state to `(chat)/layout.tsx`.

**Current state:** `(chat)/layout.tsx` is a minimal passthrough — only renders `{children}`. It can be upgraded to hold `selectedModelId` state and pass it down via context.

**Recommended approach (Claude's Discretion):** Create a `ChatModelContext` (or use URL search params for simplicity). Given the chat pages are independent routes with their own state, a context provider wrapping the chat layout is cleanest.

**Header location:** Neither `(chat)/page.tsx` nor `(chat)/threads/[id]/page.tsx` has a visible header element. The dropdown should be added as a sticky header bar at the top of both pages, or injected into the chat layout above `{children}`.

**API client needed:** Add `fetchAllowedModels()` to `frontend/lib/api/parameters.ts` (or a new `allowed-models.ts`). Add `useAllowedModels()` hook. The query key `['admin', 'allowed-models']` should be added to `query-keys.ts`.

### Anti-Patterns to Avoid
- **Rebuilding `OpenAiApi` per model:** Wasteful — share the auto-configured `OpenAiApi` bean.
- **Storing selected model in localStorage only:** State resets on refresh; use URL params or React context.
- **Making `modelId` required in `ChatQuestionRequest`:** D-15 is explicit — nullable, no validation.
- **Calling `AiParameterSetService.getActive()` on every chat request:** Only the evaluator path reads DB; chat model comes from request body (D-10).
- **Throwing HTTP 400 for unrecognized modelId:** D-12 says warn + fallback.

## Common Pitfalls

### Pitfall 1: `spring.ai.openai.chat.options.model` Conflict
**What goes wrong:** The auto-configured `OpenAiChatModel` uses `spring.ai.openai.chat.options.model` as the default model. If this YAML key exists and points to a model not in 9router's catalog, the default ChatClient (created by autoconfiguration) will fail.
**Why it happens:** Spring AI autoconfigures a `ChatModel` bean using this property. If `ChatClientConfig` also creates models, there can be two conflicting beans.
**How to avoid:** Since `ChatClientConfig` will now create the `Map<String, ChatClient>` and NOT the single `ChatClient chatClient` bean, remove the old `@Bean ChatClient chatClient(...)` from `ChatClientConfig`. The auto-configured `openAiChatModel` bean will still exist but won't be directly injected into services anymore — services use the Map instead. [ASSUMED — Spring AI 2.0.0-M4 behavior]

### Pitfall 2: `LlmSemanticEvaluator` Constructor Uses `ChatClient.Builder`
**What goes wrong:** `LlmSemanticEvaluator` currently uses constructor injection with `ChatClient.Builder chatClientBuilder` and calls `chatClientBuilder.build()` internally. After the refactor, it should not auto-build from the default Spring AI ChatClient.Builder — it needs to use the Map lookup.
**Why it happens:** The evaluator was originally built before the model catalog existed.
**How to avoid:** Change constructor to inject `Map<String, ChatClient>` and `AiModelProperties` (and `AiParameterSetService`). Remove the `ChatClient.Builder` injection. [VERIFIED: LlmSemanticEvaluator.java lines 39-42]

### Pitfall 3: `AiParameterSet.evaluatorModel` May Be Null
**What goes wrong:** The DB column `evaluatorModel` can be null for parameter sets created before this phase. Null-checking before Map.get() is mandatory.
**Why it happens:** The column was added in Phase 5 but may not be populated for all existing records.
**How to avoid:** Use `Optional` chain — `getActive().map(ps -> ps.getEvaluatorModel()).filter(Objects::nonNull).orElse(modelProperties.evaluatorModel())`. [VERIFIED: AiParameterSetService.java — getActive() returns Optional<AiParameterSet>]

### Pitfall 4: Frontend `CreateAiParameterSetRequest` Missing Model Fields
**What goes wrong:** The TypeScript type `CreateAiParameterSetRequest` only has `name` and `content` — no `chatModel` or `evaluatorModel`. The create form won't be able to set model dropdowns.
**Why it happens:** The type was defined before this phase was scoped.
**How to avoid:** Add `chatModel?: string` and `evaluatorModel?: string` to `CreateAiParameterSetRequest` in `types/api.ts`. Update `createParameterSet()` API call. Also update backend `CreateAiParameterSetRequest` DTO to accept these fields. [VERIFIED: types/api.ts lines 238-241]

### Pitfall 5: Map Lookup Returns Null If Model Catalog Is Empty
**What goes wrong:** If `app.ai.models` is empty or misconfigured, the Map is empty and any lookup throws NPE.
**Why it happens:** YAML misconfiguration or missing env var fallback.
**How to avoid:** Add a startup validation (`@PostConstruct` or `ApplicationRunner`) that logs a warning if the Map is empty. Ensure `app.ai.chat-model` default is always in the model list.

### Pitfall 6: Chat Header Dropdown State Not Persisted Across Route Transitions
**What goes wrong:** User selects model on `/` (chat home), then gets redirected to `/threads/{id}` — the selection is lost because it was in local state.
**Why it happens:** Next.js routes are separate components with independent state.
**How to avoid:** Lift model selection state to the `(chat)/layout.tsx` via React context, or use a URL search param `?model=claude-sonnet-4-6` that survives navigation.

## Open Questions

1. **Is `OpenAiApi` directly injectable from Spring AI autoconfiguration?**
   - What we know: Spring AI creates an `OpenAiApi` bean; `OpenAiChatModel` takes it as constructor arg in non-Spring usage.
   - What's unclear: Whether the auto-configured `OpenAiApi` bean is publicly exposed or wrapped. In some Spring AI versions it's private to the autoconfigure module.
   - Recommendation: If `OpenAiApi` is not injectable, fall back to constructing it manually: `new OpenAiApi(baseUrl, apiKey)` reading from `@Value` injected properties.

2. **Spring AI 2.0.0-M4 `OpenAiChatModel` constructor signature**
   - What we know: The constructor `OpenAiChatModel(OpenAiApi, OpenAiChatOptions)` exists in Spring AI 1.x.
   - What's unclear: Whether 2.0.0-M4 changed the constructor or requires a builder.
   - Recommendation: Verify by checking the actual Spring AI 2.0.0-M4 JAR in the Gradle cache, or run a quick spike. If constructor changed, use `OpenAiChatModel.builder()` if available.

3. **Does `(chat)/layout.tsx` support React context providers?**
   - What we know: It's currently a server component (no `'use client'` directive).
   - What's unclear: Whether it needs to become a client component to hold state.
   - Recommendation: Convert to `'use client'` and add context, OR use URL search params (simpler, no client component upgrade needed).
