@ExceptionHandler(DataAccessException.class)
public ProblemDetail handleDataAccessException(DataAccessException ex, HttpServletRequest request) {

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP request → GlobalExceptionHandler | Error detail from exception messages may contain sensitive DB schema or internal paths |
| User input → Log output | title, url fields from request body can contain CRLF injection payloads |
| application.yaml → AppProperties | Config values are server-controlled; no user input flows here |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-4.1-01-01 | Information Disclosure | GlobalExceptionHandler — DataAccessException detail | mitigate | Prod profile replaces cause with "Failure during data access"; dev receives full detail. Implemented in Handler A. |
| T-4.1-01-02 | Information Disclosure | GlobalExceptionHandler — generic Exception catch-all | mitigate | Prod profile replaces message with "Unexpected runtime exception". Implemented in Handler D. |
| T-4.1-01-03 | Tampering | IngestionAdminController — CRLF in title/url log fields | mitigate | CrlfSanitizer.sanitize() applied to both title and url before log interpolation. |
| T-4.1-01-04 | Information Disclosure | LoggingAspect @AfterThrowing — exception cause in non-dev | mitigate | In non-dev profiles logs only `e.getCause().getClass().getSimpleName()`, not full message. Full stack trace only in dev. |
| T-4.1-01-05 | Denial of Service | AsyncUncaughtExceptionHandler — log flooding from repeated async failures | accept | Handler logs at ERROR (not FATAL). Log volume is bounded by retry policy on ingestion jobs. Acceptable for v1. |
