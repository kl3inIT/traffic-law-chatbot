# Phase 04.1: Code Review Report

## Summary

Phase 4.1 delivers AOP logging hardening, batch ingestion, and the trust-policy CRUD stack. The overall structure is sound: CRLF sanitization is correctly applied to all user-supplied log values, the trust-tier invariant (additive metadata only, no bypass of approval/trusted/active gate) is well-enforced and documented, and the `GlobalExceptionHandler` correctly masks internal details behind the `prod` profile guard.

Two issues block a clean test run: a compile error in `GlobalExceptionHandlerTest` caused by a non-existent constructor, and a silent logic error in `AsyncConfig` that creates two separate executor instances instead of one. Four warnings cover a data-leakage risk in the batch endpoint, a controller–repository layering violation, and dropped fields on the batch DTO. Three info items cover a duplicate dependency declaration, a redundant JPA `save()` call, and the missing no-arg-constructor footgun in `GlobalExceptionHandler`.

## Critical Issues

### CR-01: `GlobalExceptionHandlerTest` calls a non-existent `NoResourceFoundException` constructor — compile failure

**File:** `src/test/java/com/vn/traffic/chatbot/common/error/GlobalExceptionHandlerTest.java:99-100`

**Issue:** The test constructs `NoResourceFoundException` with three arguments:
```java
throw new NoResourceFoundException(
    org.springframework.http.HttpMethod.GET, "/test/no-resource", "/test/no-resource");
```
`NoResourceFoundException` (Spring 6.1+) only exposes a two-argument constructor: `(HttpMethod httpMethod, String resourcePath)`. The three-argument form does not exist. This is a compile-time error that will cause the entire test class to fail to compile, blocking the test suite.

**Fix:**
```java
throw new NoResourceFoundException(
    org.springframework.http.HttpMethod.GET, "/test/no-resource");
```

### CR-02: `AsyncConfig.getAsyncExecutor()` creates a second, disconnected executor rather than reusing the `@Bean`

**File:** `src/main/java/com/vn/traffic/chatbot/common/config/AsyncConfig.java:42-52`

**Issue:** The comment on `getAsyncExecutor()` says "Delegate to the named bean" but the implementation creates a brand-new `ThreadPoolTaskExecutor` with hardcoded parameters (core=4, max=10, queue=100), independent of the `@Bean("ingestionExecutor")` instance. This produces two separate executors:

1. The named `ingestionExecutor` `@Bean`, using `ThreadPoolTaskExecutorBuilder` defaults (typically core=1, max=`Integer.MAX_VALUE`).
2. A second unnamed executor returned from `getAsyncExecutor()`, which the `AsyncUncaughtExceptionHandler` is actually bound to.

As a result, `@Async("ingestionExecutor")` tasks run on the named bean (pool size unknown, no uncaught handler coverage), while `@Async` tasks without an explicit name run on the second unnamed executor (uncaught handler covered). The intent — a single pool with the uncaught handler — is not achieved.

**Fix:** Inject the named executor bean and return it from `getAsyncExecutor()`:
```java
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {

    private static final Logger LOG = LoggerFactory.getLogger(AsyncConfig.class);

    @Bean(name = "ingestionExecutor")
    public ThreadPoolTaskExecutor ingestionExecutor(ThreadPoolTaskExecutorBuilder builder) {
        return builder
                .corePoolSize(4)
                .maxPoolSize(10)
                .queueCapacity(100)
                .threadNamePrefix("ingestion-")
                .build();
    }

    @Override
    public Executor getAsyncExecutor() {
        // Return the same named bean — Spring resolves this safely
        // because @Bean methods on @Configuration classes are proxied.
        return ingestionExecutor(null); // or inject via @Autowired field
    }
    ...
}
```
The cleanest approach is to inject `ThreadPoolTaskExecutor ingestionExecutor` as a field via `@Autowired` (or constructor injection) and return it directly from `getAsyncExecutor()`.

## Warnings

### WR-01: Batch endpoint exposes raw exception messages to API callers

**File:** `src/main/java/com/vn/traffic/chatbot/ingestion/api/IngestionAdminController.java:75-78`

**Issue:** The `catch (Exception ex)` block at lines 75-78 puts `ex.getMessage()` directly into the `BatchImportItemResult.errorMessage` field, which is returned to the client in the 200 response body. In non-dev environments this can expose internal exception details (SQL messages, stack frame info from chained exceptions, file paths, etc.) that should be masked for callers.

**Fix:** Classify the exception type before serialising the message. At minimum, only expose the message for known `AppException` instances; use a generic message for all others:
```java
} catch (AppException ex) {
    results.add(new BatchImportItemResult(
            item.url(), "ERROR", null, null, ex.getMessage()));
    rejected++;
} catch (Exception ex) {
    log.error("Unexpected error processing batch item url={}: {}", sanitizedUrl, ex.getMessage(), ex);
    results.add(new BatchImportItemResult(
            item.url(), "ERROR", null, null, "Ingestion failed"));
    rejected++;
}
```

### WR-02: `IngestionAdminController` directly injects and queries `KbSourceRepository` — layering violation

**File:** `src/main/java/com/vn/traffic/chatbot/ingestion/api/IngestionAdminController.java:32,63-64`

**Issue:** The controller injects `KbSourceRepository` and performs a duplicate-URL check inline (lines 63-64). This logic belongs in `IngestionService`: the controller should not know about the repository layer, and moving the check into the service ensures it participates in the service's transaction boundary and is testable in isolation.

**Fix:** Move the `findByOriginValue` duplicate check into `IngestionService.submitUrl()` or a dedicated `checkDuplicate()` method, and have the service throw a known `AppException(ErrorCode.DUPLICATE_SOURCE, ...)` on conflict. The controller then catches it via `GlobalExceptionHandler` or handles the `AppException` return path without holding a repository reference.

### WR-03: `BatchImportItemRequest.sourceType` and `trustCategory` fields are declared but silently discarded

**File:** `src/main/java/com/vn/traffic/chatbot/ingestion/api/IngestionAdminController.java:69`
**Supporting file:** `src/main/java/com/vn/traffic/chatbot/ingestion/api/dto/BatchImportItemRequest.java:10-12`

**Issue:** `BatchImportItemRequest` declares `@NotNull SourceType sourceType` and `TrustTier trustCategory`, but the controller constructs `UrlSourceRequest(item.url(), item.title(), null, null)`, passing `null` for both fields. The caller's `sourceType` and `trustCategory` are silently dropped. Because `sourceType` is annotated `@NotNull`, the caller is forced to supply it, yet it has no effect.

**Fix:** Either (a) pass `item.sourceType()` and `item.trustCategory()` to `UrlSourceRequest` (and ensure `UrlSourceRequest` accepts them), or (b) remove the fields from `BatchImportItemRequest` if they are intentionally not used at this stage. Using `@NotNull` on a field that is then ignored is misleading and will cause unnecessary validation failures for callers that do not supply it.

### WR-04: `GlobalExceptionHandler` no-arg constructor allows silent null `Environment` in production

**File:** `src/main/java/com/vn/traffic/chatbot/common/error/GlobalExceptionHandler.java:39-41`

**Issue:** The no-arg constructor sets `this.env = null` and is documented as "for standalone MockMvc tests". If this bean is ever instantiated by Spring without the `Environment` argument (e.g., via reflection, a proxy misconfiguration, or a test setup that does not exclude the class from Spring context bootstrapping), `isProdProfile()` returns `false`, causing full internal exception detail to be exposed in production. The existence of a public no-arg constructor on a Spring-managed `@RestControllerAdvice` is a latent risk — Spring prefers the most-specific constructor but the presence of the no-arg form provides an escape hatch that can be invoked unexpectedly.

**Fix:** Remove the no-arg constructor and annotate it only for test use if truly needed, or use a default `Environment` stub. Tests that need `GlobalExceptionHandler` without Spring context should supply a mock `Environment` explicitly (as `GlobalExceptionHandlerTest` already does via `new GlobalExceptionHandler(env)` with a Mockito mock):
```java
// Remove the no-arg constructor entirely.
// Test code already uses: new GlobalExceptionHandler(env) with mock env.
```

### IN-02: `SourceTrustPolicyService.update()` calls `save()` on an already-managed entity

**File:** `src/main/java/com/vn/traffic/chatbot/source/service/SourceTrustPolicyService.java:61`

**Issue:** The entity returned by `findById()` inside a `@Transactional` method is a JPA-managed instance. Mutating its fields and then calling `trustPolicyRepository.save(policy)` is redundant — the dirty-checking mechanism will persist the changes at transaction commit without the explicit `save()` call. This is not a bug, but it establishes a pattern that may mislead future maintainers into thinking explicit `save()` is required for managed entities.

**Fix:** Remove the explicit `save()` call and return `toResponse(policy)` directly:
```java
public TrustPolicyResponse update(UUID id, TrustPolicyRequest request) {
    SourceTrustPolicy policy = trustPolicyRepository.findById(id)
            .orElseThrow(() -> new AppException(ErrorCode.TRUST_POLICY_NOT_FOUND,
                    "Trust policy not found: " + id));
    policy.setName(request.name());
    policy.setDomainPattern(request.domainPattern());
    policy.setSourceType(request.sourceType());
    policy.setTrustTier(request.trustTier());
    policy.setDescription(request.description());
    // Dirty-checking flushes changes at commit; no explicit save() needed.
    log.info("Trust policy updated: id={}, name='{}', tier={}", id, policy.getName(), policy.getTrustTier());
    return toResponse(policy);
}
```

### IN-03: `AsyncConfigTest` does not verify `getAsyncExecutor()` returns a non-null executor

**File:** `src/test/java/com/vn/traffic/chatbot/common/config/AsyncConfigTest.java`

**Issue:** The test verifies `AsyncConfig` implements `AsyncConfigurer` and that `getAsyncUncaughtExceptionHandler()` is non-null, but does not verify `getAsyncExecutor()`. Given that CR-02 above identifies a logical error in `getAsyncExecutor()`, a test assertion on it would have caught the problem earlier.

**Fix:** Add a test case:
```java
@Test
void getAsyncExecutorReturnsNonNull() {
    AsyncConfigurer config = new AsyncConfig();
    assertThat(config.getAsyncExecutor())
            .as("getAsyncExecutor() must return a non-null Executor")
            .isNotNull();
}
```
Note that after fixing CR-02, the `@Bean` dependency injection approach means `getAsyncExecutor()` will require the Spring context to resolve; adjust the test to use `@SpringBootTest` or mock the dependency as appropriate.
