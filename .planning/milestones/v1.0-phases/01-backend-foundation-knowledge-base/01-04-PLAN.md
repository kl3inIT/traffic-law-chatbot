<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| RetrievalPolicy -> VectorStore | Filter expression must be immutable and enforced at every retrieval call |
| ChunkInspectionService -> PostgreSQL | JDBC queries on kb_vector_store must use parameterized queries to prevent SQL injection |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-04-01 | Elevation of Privilege | RetrievalPolicy filter expression | mitigate | RETRIEVAL_FILTER is a static final constant; callers cannot override it. Phase 2 chat pipeline MUST use RetrievalPolicy.buildRequest() exclusively. Verified by RetrievalPolicy tests 1-3. |
| T-04-02 | Injection | ChunkInspectionService JDBC queries | mitigate | All WHERE clauses use PreparedStatement parameter binding (jdbcTemplate.query with PreparedStatementSetter), not string concatenation. |
| T-04-03 | Tampering | ChunkMetadataUpdater SQL | mitigate | UPDATE uses parameterized sourceId; jsonb_set path is hardcoded ('\{trusted\}', '\{active\}'), not user-supplied. |
| T-04-04 | Information Disclosure | ChunkDetailResponse.content | accept | Chunk content returned to admin callers only. No auth in Phase 1; auth added in Phase 4. Admin API is internal-only by deployment convention. |
| T-04-05 | Denial of Service | getIndexSummary five COUNT queries | accept | Counts run sequentially on bounded PostgreSQL table. Add index on metadata->>'approvalState' in a future migration if performance degrades. |
</threat_model>
