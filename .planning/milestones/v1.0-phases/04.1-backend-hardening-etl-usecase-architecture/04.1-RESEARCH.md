### Pattern 1: LoggingAspect (Track A — D-01)

**What:** AOP aspect activated via `@Configuration` + `@Bean` gated by `@Profile("dev")` for the Around trace. AfterThrowing advice is always-on.

**JHipster reference:** `D:/DTH/demo-next-frontend/.../LoggingAspect.java` — adapt package names, remove JHipsterConstants dependency.

**Key adaptation from JHipster reference:**
- JHipster checks `env.acceptsProfiles(Profiles.of(JHipsterConstants.SPRING_PROFILE_DEVELOPMENT))` inside the advice. Our approach: register the bean only in dev profile for the Around trace, but keep AfterThrowing in a separate always-active aspect (or use the env check inside the same class).
- Simpler approach (Claude's discretion): one class, `@AfterThrowing` without profile guard, `@Around` with `if (env.acceptsProfiles(Profiles.of("dev")))` guard inside the method body.
- Pointcut covers `common.aop..`, `ingestion..`, `source..`, `chat..`, `retrieval..`, `parameter..`, `chunk..` packages within `com.vn.traffic.chatbot`.

```java
// Source: JHipster reference adapted for this project
@Aspect
@Component
public class LoggingAspect {

    private final Environment env;

    public LoggingAspect(Environment env) { this.env = env; }

    @Pointcut("within(@org.springframework.stereotype.Repository *)" +
              " || within(@org.springframework.stereotype.Service *)" +
              " || within(@org.springframework.web.bind.annotation.RestController *)")
    public void springBeanPointcut() {}

    @Pointcut("within(com.vn.traffic.chatbot.ingestion..*)" +
              " || within(com.vn.traffic.chatbot.source..*)" +
              " || within(com.vn.traffic.chatbot.chat..*)" +
              " || within(com.vn.traffic.chatbot.retrieval..*)" +
              " || within(com.vn.traffic.chatbot.parameter..*)" +
              " || within(com.vn.traffic.chatbot.chunk..*)")
    public void applicationPackagePointcut() {}

    @AfterThrowing(pointcut = "applicationPackagePointcut() && springBeanPointcut()", throwing = "e")
    public void logAfterThrowing(JoinPoint joinPoint, Throwable e) {
        // Always active — exception traceability in all profiles
        if (env.acceptsProfiles(Profiles.of("dev"))) {
            logger(joinPoint).error("Exception in {}() with cause='{}' exception='{}'",
                joinPoint.getSignature().getName(), e.getCause(), e.getMessage(), e);
        } else {
            logger(joinPoint).error("Exception in {}() with cause={}",
                joinPoint.getSignature().getName(),
                e.getCause() != null ? e.getCause().getClass().getSimpleName() : "NULL");
        }
    }

    @Around("applicationPackagePointcut() && springBeanPointcut()")
    public Object logAround(ProceedingJoinPoint joinPoint) throws Throwable {
        // Dev-profile-only enter/exit tracing
        if (env.acceptsProfiles(Profiles.of("dev")) && logger(joinPoint).isDebugEnabled()) {
            logger(joinPoint).debug("Enter: {}() args={}", joinPoint.getSignature().getName(),
                Arrays.toString(joinPoint.getArgs()));
        }
        Object result = joinPoint.proceed();
        if (env.acceptsProfiles(Profiles.of("dev")) && logger(joinPoint).isDebugEnabled()) {
            logger(joinPoint).debug("Exit: {}() result={}", joinPoint.getSignature().getName(), result);
        }
        return result;
    }

    private Logger logger(JoinPoint jp) {
        return LoggerFactory.getLogger(jp.getSignature().getDeclaringTypeName());
    }
}
```

[VERIFIED: JHipster reference at D:/DTH/demo-next-frontend]

**Registration:** `@Component` with `@Profile("dev")` does NOT work here because AfterThrowing must be always-on. Use `@Component` (no profile) and guard `@Around` body with env check. The `@Aspect` + `@Component` combo is picked up by Spring's AOP auto-proxy configuration.

**Pitfall — AspectJ not on classpath:** Spring Boot 4.x uses Spring AOP (proxy-based) by default. `@Aspect` + `@EnableAspectJAutoProxy` (or `spring.aop.auto=true` which is the default) works without AspectJ weaving. No `aspectjweaver` jar is required as a compile dependency because `spring-boot-starter-webmvc` transitively includes `spring-aop` which contains the annotation support.

### Pattern 2: AsyncConfig with AsyncConfigurer (Track A — D-04)

**What:** Implement `AsyncConfigurer` on `AsyncConfig`, add `getAsyncUncaughtExceptionHandler()` returning a custom handler that logs method name, parameters, and exception.

**JHipster reference:** `AsyncConfiguration.java` uses `ExceptionHandlingAsyncTaskExecutor` (a JHipster wrapper). We do NOT use JHipster's wrapper — implement the handler directly.

**Current state:** `AsyncConfig` declares `@Bean(name = "ingestionExecutor")` using `ThreadPoolTaskExecutorBuilder`. The pool settings (`core-size: 4`, `max-size: 10`, `queue-capacity: 100`, `thread-name-prefix: ingestion-`) are read from `application.yaml` `spring.task.execution.*` by the builder automatically.

**Upgrade pattern:**

```java
// Source: JHipster reference adapted
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {

    private static final Logger LOG = LoggerFactory.getLogger(AsyncConfig.class);

    @Bean(name = "ingestionExecutor")
    public ThreadPoolTaskExecutor ingestionExecutor(ThreadPoolTaskExecutorBuilder builder) {
        return builder.build(); // pool config still from application.yaml
    }

    @Override
    public AsyncUncaughtExceptionHandler getAsyncUncaughtExceptionHandler() {
        return (ex, method, params) -> LOG.error(
            "Uncaught async exception in {}() params={}: {}",
            method.getName(), Arrays.toString(params), ex.getMessage(), ex);
    }
}
```

[VERIFIED: JHipster reference + current AsyncConfig.java]

**Important:** `getAsyncUncaughtExceptionHandler()` from `AsyncConfigurer` applies to Spring-managed `@Async` methods using the DEFAULT executor, not necessarily the named `ingestionExecutor` bean. Verify which executor `IngestionOrchestrator.runPipeline()` uses — if it uses `@Async("ingestionExecutor")`, the uncaught handler from `AsyncConfigurer` does NOT automatically apply. In that case, wrapping the executor in a decorator that catches and logs exceptions is the correct approach.

**What:** Typed bean binding all `app.*` YAML keys. Current `app.*` usage:

- `ChatService`: `${app.chat.retrieval.top-k:5}` and `${app.chat.grounding.limited-threshold:2}` (two `@Value`)

**Current app.* YAML structure:**
```yaml
app:

    grounding:
      limited-threshold: 2
    case-analysis:
      max-clarifications: 2
```

**JHipster reference:** `ApplicationProperties.java` is essentially empty — the pattern is the class + `@ConfigurationProperties(prefix = "application")`. Our class is richer.

### Pattern 4: GlobalExceptionHandler Hardening (Track A — D-03)

**Current state:** Handles `MethodArgumentNotValidException`, `AppException`, `MethodArgumentTypeMismatchException`, `Exception` (generic catch-all). Uses ProblemDetail RFC7807 throughout.

**Missing handlers to add:**
1. `DataAccessException` → HTTP 500, prod-masked detail
2. `ConstraintViolationException` (jakarta.validation) → HTTP 400
3. `NoHandlerFoundException` or `NoResourceFoundException` (Spring MVC 6+) → HTTP 404 for unmapped paths
4. Profile-aware detail in generic catch-all (already present as generic message, refine to add prod mask)

**Profile detection pattern in handler:**

```java
@Autowired
private Environment env;

private boolean isProdProfile() {
    return env.acceptsProfiles(Profiles.of("prod"));
}
```

**Spring MVC unmapped path 404:** In Spring Boot 4 / Spring MVC 6, `NoHandlerFoundException` is replaced by `NoResourceFoundException` for static resources. For REST API 404s on unmapped paths, configure:
```yaml
spring:
  mvc:
    throw-exception-if-no-handler-found: true
  web:
    resources:
      add-mappings: false
```
Then handle `NoHandlerFoundException`. Alternatively, override `configureHandlerExceptionResolvers` — but the simpler approach is the YAML property + handler.

**Actual class for unmapped paths in Spring Boot 4:** `org.springframework.web.servlet.resource.NoResourceFoundException` (HTTP 404 by default). Add an explicit handler to produce consistent ProblemDetail:

```java
@ExceptionHandler(NoResourceFoundException.class)
public ProblemDetail handleNoResource(NoResourceFoundException ex, HttpServletRequest request) {
    ProblemDetail problem = ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, "Resource not found");
    problem.setInstance(URI.create(request.getRequestURI()));
    return problem;
}
```

[ASSUMED — Spring Boot 4.0.5 exact exception class for unmapped paths needs verification; may be `NoHandlerFoundException` or `NoResourceFoundException`]

**ConstraintViolationException:** `jakarta.validation.ConstraintViolationException` is thrown when `@Validated` is used on a service/controller and method-level constraints fail (distinct from `MethodArgumentNotValidException` which covers request body `@Valid`).

```java
@ExceptionHandler(ConstraintViolationException.class)
public ProblemDetail handleConstraintViolation(ConstraintViolationException ex, HttpServletRequest request) {
    Map<String, String> errors = ex.getConstraintViolations().stream()
        .collect(Collectors.toMap(
            v -> v.getPropertyPath().toString(),
            ConstraintViolation::getMessage,
            (a, b) -> a));
    ProblemDetail problem = ProblemDetail.forStatusAndDetail(HttpStatus.BAD_REQUEST, "Constraint violation");
    problem.setInstance(URI.create(request.getRequestURI()));
    problem.setProperty("errors", errors);
    return problem;
}
```

[VERIFIED: spring-boot-starter-validation is on classpath per build.gradle]

### Pattern 5: CRLF Log Injection Prevention (Track A — D-06)

**What:** Prevent log forging by stripping or replacing `\r` and `\n` characters in user-supplied strings before they appear in log statements.

**Standard approach:** Utility method that sanitizes a string value before logging:

```java
// Source: OWASP log injection prevention pattern [ASSUMED — OWASP standard]
private static String sanitizeForLog(String value) {
    if (value == null) return "null";
    return value.replace('\r', '_').replace('\n', '_');
}
```

This is applied at log callsites for user-supplied inputs (URL, title, sourceId from requests). The AOP aspect does not need modification — it logs method names and Spring-managed objects, not raw user input.

**Where to apply:**
- `IngestionAdminController`: `log.info("Upload source request: title={}", metadata.getTitle())` → sanitize title
- `IngestionAdminController`: `log.info("URL ingest request: url={}", req.url())` → sanitize URL
- New batch endpoint logging
- Any other controller/service log where user-supplied strings are interpolated

**Alternative (logback pattern):** Configure a Logback PatternLayout that escapes CRLF. However, adding a utility method is simpler, targeted, and does not require Logback config changes.

### Pattern 7: Source Trust Policy DB (Track B — D-10, D-11)

**What:** New `source_trust_policy` table. Trust tier is metadata — separate lookup table where each row defines a tier rule (by domain pattern or source type), and `KbSource` can optionally reference a policy or carry the tier directly.

**Two design options:**

| Option | Schema | Tradeoff |
|--------|--------|----------|
| A: Tier on KbSource | Add `trust_tier VARCHAR(50)` column to `kb_source` | Simple, no join, but trust classification is per-source not per-domain |
| B: Separate policy table | New `source_trust_policy` table, optional FK or domain-pattern lookup | Runtime admin-configurable domain rules, matches D-10 "runtime CRUD without redeployment" |

**Recommendation — Option B (separate table, matches D-10 intent):**

```xml
<!-- 007-source-trust-policy-schema.xml -->
<createTable tableName="source_trust_policy">
    <column name="id" type="UUID" defaultValueComputed="gen_random_uuid()">
        <constraints primaryKey="true" nullable="false"/>
    </column>
    <column name="name" type="VARCHAR(255)">
        <constraints nullable="false"/>
    </column>
    <column name="domain_pattern" type="VARCHAR(500)"/>
    <!-- e.g., "*.moj.gov.vn", "thuvienphapluat.vn" -->
    <column name="source_type" type="VARCHAR(50)"/>
    <!-- optional filter: apply only to PDF, URL, etc. -->
    <column name="trust_tier" type="VARCHAR(50)">
        <constraints nullable="false"/>
    </column>
    <!-- PRIMARY | SECONDARY | MANUAL_REVIEW -->
    <column name="description" type="TEXT"/>
    <column name="created_at" type="TIMESTAMPTZ" defaultValueComputed="now()">
        <constraints nullable="false"/>
    </column>
    <column name="updated_at" type="TIMESTAMPTZ" defaultValueComputed="now()">
        <constraints nullable="false"/>
    </column>
</createTable>
```

Additionally, add `trust_tier` column to `kb_source` to store the resolved tier per source (denormalized for query efficiency):

```xml
<!-- Part of same changeset or 007 -->
<addColumn tableName="kb_source">
    <column name="trust_tier" type="VARCHAR(50)" defaultValue="MANUAL_REVIEW">
        <constraints nullable="false"/>
    </column>
</addColumn>
```

This allows: admin defines domain policies in `source_trust_policy`; when a source is created (especially via batch import), the service looks up a matching policy and sets `kb_source.trust_tier`. Admin can also override per-source manually via the REST API.

**Entity placement:** `source/domain/SourceTrustPolicy.java` — consistent with existing `KbSource` in `source/domain/`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| AOP method logging | Custom try/catch in every method | Spring AOP `@Aspect` + `@Around`/`@AfterThrowing` | Spring AOP handles proxy interception; hand-rolling loses transitive call coverage |
| Async exception surfacing | Custom try/catch inside `@Async` methods | `AsyncConfigurer.getAsyncUncaughtExceptionHandler()` | Unchecked exceptions from `@Async` are silently swallowed without this |
| Config property binding | Scattered `@Value` annotations | `@ConfigurationProperties` typed bean | Type-safe, IDE-completable, testable with `@ConfigurationPropertiesTest` |
| 409 dedup logic | Custom URL hash table | Query `KbSourceRepository.findByOriginValue(url)` | JPA already owns the source table; no auxiliary cache needed |

## Common Pitfalls

### Pitfall 1: @Async Uncaught Handler Scope

**What goes wrong:** `AsyncConfigurer.getAsyncUncaughtExceptionHandler()` only applies to `@Async` methods using the default task executor, not named executors like `@Async("ingestionExecutor")`.

**Why it happens:** Spring invokes the uncaught handler only for the executor returned by `AsyncConfigurer.getAsyncExecutor()`. Named beans are independent.

**How to avoid:** Two options:
1. Override `getAsyncExecutor()` in `AsyncConfig` to return the `ingestionExecutor`, making it the default. Named references then also use the same handler.
2. Wrap the `ThreadPoolTaskExecutor` in a decorator that adds try/catch logging before delegation.

**Check:** Read `IngestionOrchestrator.runPipeline()` — if annotated `@Async("ingestionExecutor")`, option 1 or 2 applies. If `@Async` with no name, the `AsyncConfigurer` handler applies directly.

[VERIFIED: AsyncConfig.java inspected — current bean is named "ingestionExecutor"]

### Pitfall 2: @Profile on Aspect Bean vs @Profile Guard Inside Method

**What goes wrong:** Annotating the whole `LoggingAspect` class with `@Profile("dev")` would disable `@AfterThrowing` in non-dev profiles (breaking D-01 requirement for always-on exception logging).

**How to avoid:** Do NOT put `@Profile("dev")` on the class. Use `env.acceptsProfiles(Profiles.of("dev"))` guard INSIDE the `@Around` advice body only.

### Pitfall 3: ConstraintViolationException Import Ambiguity

**What goes wrong:** Both `jakarta.validation.ConstraintViolationException` and `org.hibernate.exception.ConstraintViolationException` exist. The handler must catch the Jakarta one (validation) not the Hibernate one (DB constraint, which is a `DataAccessException`).

**How to avoid:** Explicit import: `import jakarta.validation.ConstraintViolationException;`. The Hibernate DB constraint exception will be caught by the `DataAccessException` handler (since Hibernate wraps DB errors).

### Pitfall 4: Batch Import Partial Failure Handling

**What goes wrong:** If the batch coordinator runs all items in a single `@Transactional` method, one 409 duplicate rolls back all successful items.

**How to avoid:** The batch endpoint must NOT be transactional as a single unit. Each item creates its own transaction (delegating to `IngestionService.submitUrl()` which is `@Transactional`). The controller collects results per item.

### Pitfall 5: LoggingAspect Infinite Recursion

**What goes wrong:** If the aspect's pointcut matches the aspect class itself or other AOP infrastructure, infinite recursion occurs.

**How to avoid:** Scope the `applicationPackagePointcut` to specific domain packages (ingestion, source, chat, retrieval, parameter, chunk) and explicitly exclude the `common.aop` package from the pointcut.

### Pitfall 6: trust_tier Column Migration — Nullable Default

**What goes wrong:** Adding `trust_tier` to `kb_source` with `NOT NULL` constraint requires a default value that covers all existing rows.

**How to avoid:** Use `defaultValue="MANUAL_REVIEW"` in the Liquibase changeset (consistent with D-11: unclassified sources are MANUAL_REVIEW). Existing rows automatically get MANUAL_REVIEW tier.

### Pitfall 7: Next.js AGENTS.md Warning

**What goes wrong:** The frontend has `frontend/AGENTS.md` warning that "This version has breaking changes" and instructs reading `node_modules/next/dist/docs/` before writing code.

**How to avoid:** The trust policy admin page MUST follow the exact same patterns as existing admin pages (sources/page.tsx, parameters/page.tsx). Do not introduce new Next.js features or APIs not already used in the codebase. The existing `useQuery`/`useMutation` + `apiGet`/`apiPost` + `DataTable`/`form` patterns are safe to replicate.

   - What we know: Spring MVC 6 replaced `NoHandlerFoundException` for static resources with `NoResourceFoundException`. REST unmapped paths may differ.
   - What's unclear: Whether Boot 4.0.5 auto-configures `NoHandlerFoundException` for `/api/**` paths by default.
   - Recommendation: Test with a curl to an unmapped path and observe actual exception class in logs. Then add the correct handler.
   - **RESOLVED (04.1-01-T3):** Plan A T3 includes a conditional fallback — if `NoResourceFoundException` is not on the classpath, fall back to `NoHandlerFoundException`. Both are handled to cover the Spring Boot 4.0.5 behaviour.

3. **Trust policy DB schema — FK from kb_source to source_trust_policy or domain-match at service level?**
   - What we know: D-10 says runtime CRUD. D-11 says trust tier is metadata.
   - What's unclear: Whether trust tier is stored as a denormalized column on `kb_source` (set at creation) or computed via policy lookup at query time.
   - Recommendation: Denormalized column on `kb_source` (set during batch import or source creation) + separate `source_trust_policy` reference table for admin-defined domain rules.    - **RESOLVED (04.1-02-T1):** Denormalized `trust_tier` column on `kb_source` (set at batch import / source creation) + separate `source_trust_policy` reference table for admin-managed domain rules. Changeset 007 adds both.
