## Section 3 — Security Note: Why `RetrievalPolicy.RETRIEVAL_FILTER` Must Remain Hardcoded

`RetrievalPolicy.RETRIEVAL_FILTER` contains the expression:

```
approvalState == 'APPROVED' && trusted == 'true' && active == 'true'
```

This filter is applied to every vector store similarity search, and it enforces three orthogonal trust gates simultaneously:

1. **`approvalState == 'APPROVED'`** — Only chunks from sources that have been explicitly approved by an admin can enter retrieval. Sources in `PENDING` or `REJECTED` states are excluded.
2. **`trusted == 'true'`** — Only chunks from sources designated as trusted knowledge bases (official legal documents, regulatory publications) are included. Untrusted sources are excluded regardless of approval state.
3. **`active == 'true'`** — Only chunks from currently active sources are included. Deactivated or archived sources cannot contribute to answers.

Making this filter admin-configurable would mean:
- An admin could remove the `trusted == 'true'` condition, allowing unvetted community contributions into legal answers.
- An admin could remove the `approvalState == 'APPROVED'` condition, allowing freshly ingested, unreviewed content into answers before an admin review cycle completes.
- The filter expression could be set to an empty string or always-true expression, bypassing all three gates simultaneously.

For a legal advice context (Vietnam traffic law), any failure in this gate could produce answers grounded in untrustworthy or unapproved sources. The Phase 01 design decision D-13 designated this as a permanent hardcode specifically because the security implications of misconfiguration are high and the operational benefit of making it configurable is low (the gate is intentionally minimal and rarely needs adjustment).

**This hardcode is intentional, not an oversight. Do not make this configurable.**

## Section 4 — Priority Recommendations

### P1 — Next Phase (High Value, Limited Risk)

**ClarificationPolicy.toPendingFact() — Parameterize clarification question and reason strings**

- **What:** The `toPendingFact()` switch expression in `ClarificationPolicy` returns hardcoded Vietnamese clarification questions and reason strings for each fact key (`vehicleType`, `violationType`, `injuryStatus`, `alcoholStatus`, `licenseStatus`, `documentStatus`).
- **Why next phase:** These strings appear directly in the chat UI as the clarification prompts shown to users. An admin deploying for a different legal context (e.g., commercial vehicle operators, administrative procedure queries) would need to customize the question wording. Without parameterization, this requires a code change.
- **Proposed approach:** Add named keys to `AiParameterSet` YAML schema, e.g.:
  - `clarification.question.vehicleType`
  - `clarification.reason.vehicleType`
  - `clarification.question.violationType`
  - `clarification.reason.violationType`
  - ... one pair per fact key ...
  - `clarification.question.default`
  - `clarification.reason.default`
- **Read via:** `paramProvider.getString("clarification.question.vehicleType", DEFAULT_QUESTION_VEHICLE_TYPE)` — keep current strings as static fallback constants.
- **Risk:** Low — additive change to parameter schema; fallback to current strings preserves existing behavior when no parameter override is configured.

### P2 — Future Phase (Medium Value, Low Urgency)

**ChatService.containsLegalSignal() — Expose legal signal keyword list as admin-editable parameter**

- **What:** The `containsLegalSignal()` method in `ChatService` uses a hardcoded list of 15 Vietnamese legal signal keywords to determine whether a retrieved citation qualifies as a legal citation.
- **Why flagged:** The keyword list is a quality heuristic. If legal terminology evolves (new regulatory document types, abbreviations, regional naming conventions) the list becomes stale without a code change. An admin who understands legal domain terminology would benefit from being able to extend this list without a deployment.
- **Proposed approach:** Add `chat.legalSignalKeywords` to `AiParameterSet` schema as a list field. Read via `paramProvider.getList("chat.legalSignalKeywords")` with the current 15-keyword list as the built-in default. If the list from params is non-empty, use it; otherwise fall back to the built-in list.
- **Risk:** Medium — changing the keyword list affects grounding behavior. Overly broad lists increase false positives (non-legal content passing as legal). Overly narrow lists cause legal citations to be dismissed. Recommend adding a test fixture that verifies representative keywords before enabling admin editing.

### Permanent — No Action Required

**RetrievalPolicy.RETRIEVAL_FILTER — Hardcoded by design, never expose as configurable**

- This is a security boundary, not a configuration parameter. See Section 3 for full rationale.
- Any architectural proposal to make this configurable must be treated as a security-critical change requiring threat model review and explicit sign-off.

## Section 5 — Coverage Summary: AnswerCompositionPolicy

`AnswerCompositionPolicy` has **zero configurability gaps**.

All five user-facing string categories in the class are fully covered by the existing `AiParameterSet` parameter system:

| Method | Parameter Key | Fallback Constant |
|--------|---------------|-------------------|
| `getDisclaimer()` | `messages.disclaimer` | `DEFAULT_DISCLAIMER` |
| `getRefusalMessage()` | `messages.refusal` | `REFUSAL_MESSAGE` |
| `getLimitedNotice()` | `messages.limitedNotice` | `LIMITED_NOTICE` |
| `getRefusalNextSteps()` step 1 | `messages.refusalNextStep1` | `REFUSAL_NEXT_STEP_NARROW_SCOPE` |
| `getRefusalNextSteps()` step 2 | `messages.refusalNextStep2` | `REFUSAL_NEXT_STEP_NAME_DOCUMENT` |
| `getRefusalNextSteps()` step 3 | `messages.refusalNextStep3` | `REFUSAL_NEXT_STEP_VERIFY_SOURCE` |

The static constants (`public static final String`) remain in the class for backward compatibility with tests and comparison logic, not as hardcoded string injection. They serve exclusively as fallback defaults.

**Verdict:** `AnswerCompositionPolicy` is the model for how user-facing strings should be handled in this codebase. The pattern should be applied to `ClarificationPolicy.toPendingFact()` in the next phase.

### ClarificationPolicy.java — builtInRequiredFactKeys() method (lines 89–107)

```java
private Set<String> builtInRequiredFactKeys(String normalized, Map<String, String> factMap) {
    Set<String> required = new HashSet<>();
    required.add("vehicleType");     // always required (built-in path)
    required.add("violationType");   // always required (built-in path)

    if (normalized.contains("tai nạn") || normalized.contains("va chạm") || factMap.containsKey("injuryStatus")) {
        required.add("injuryStatus");
    }
    if (normalized.contains("nồng độ cồn") || normalized.contains("rượu") || normalized.contains("bia") || ...) {
        required.add("alcoholStatus");
    }
    if (normalized.contains("bằng lái") || normalized.contains("giấy phép lái xe") || ...) {
        required.add("licenseStatus");
    }
    if (normalized.contains("đăng ký xe") || normalized.contains("cà vẹt") || normalized.contains("giấy tờ") || ...) {
        required.add("documentStatus");
    }
    return required;
}
```
