<decisions>
## Implementation Decisions

### Track A — Backend operational hardening

- **D-01:** `LoggingAspect` (AOP) — enter/exit method debug tracing is **dev-profile-only** (activated only when `spring.profiles.active=dev`). Exception logging (`@AfterThrowing`) is **always active** across all profiles. Package location and exact pointcut configuration are Claude's discretion.
- **D-02:** Explicit **structured logging at key operational boundaries** — the four suggested areas are the baseline: ingestion job lifecycle (start, step transitions, completion with chunk count, failure with cause, jobId + sourceId), source approval/trust/activation events (sourceId, name, trigger), chat request lifecycle (query summary, retrieved chunk count, grounding status), vector store operations (embedding + upsert counts per job). Exact log placements are Claude's discretion — the goal is traceability for debugging, not exhaustive logging of every method.
- **D-03:** `GlobalExceptionHandler` hardening — add `DataAccessException` handler (HTTP 500, suppress cause in prod), `ConstraintViolationException` handler (HTTP 400), and unmapped-path 404 handler. **Production masking:** when `spring.profiles.active=prod`, replace DataAccessException cause with `"Failure during data access"` and generic catch-all with `"Unexpected runtime exception"`. Dev/staging profiles receive full detail.
- **D-04:** `AsyncConfig` upgraded to implement `AsyncConfigurer` with explicit `getAsyncUncaughtExceptionHandler()` to surface silent `@Async` failures from ingestion tasks. Implementation is Claude's discretion.
- **D-05:** Typed `AppProperties` `@ConfigurationProperties` bean added for `app.*` namespace. Current scatter is minimal (only `CorsConfig` and `ChatService` use `@Value` for `app.*`). Whether to migrate those two call sites to the typed bean or leave them is Claude's discretion — the bean must exist as the canonical config access point going forward.
- **D-06:** CRLF log injection prevention added to log output. Implementation approach is Claude's discretion.

### Track B — Batch ingestion + Source trust policy

- **D-07:** Phase 01.1-04 is **already executed** (commit `146a22c`, 2026-04-08 — Spring AI ETL promotion and token chunking shipped). Only task required: update ROADMAP.md checkbox for `[ ] 01.1-04-PLAN.md` → `[x]`.
- **D-08:** New batch import endpoint: `POST /api/v1/admin/ingestion/batch` accepts a list of `{url, title, sourceType, trustCategory}` objects. Each item creates one `KbSource` and triggers an async ingestion job via the existing pipeline. This reuses all existing ingestion domain behavior (SSRF checks, ParsedDocument boundary, job tracking, approval gating).
- **D-09:** Batch dedup — if a `KbSource` already exists for a given URL (any status), return HTTP 409 for that item. Per-item 409 should not abort the entire batch — report successes and failures in the response.
- **D-10:** **Source trust policy** — stored in a new DB table (`source_trust_policy` or similar), not in YAML config. Supports runtime CRUD without redeployment. REST API exposed for admin management.
- **D-11:** Trust tier model has **three tiers**: `PRIMARY` (official legal documents, official government portals), `SECONDARY` (legal reference/aggregator websites), `MANUAL_REVIEW` (unclassified or user-submitted sources). Trust tier is a metadata field on sources or trust policy entries — it does NOT bypass the existing approval/trusted/active retrieval gate. All sources still require explicit admin approval to become retrieval-eligible.
- **D-12:** **Phase 4.1 includes an admin UI screen** for trust policy management in the existing Next.js `frontend/` app (alongside other admin screens from Phase 4). Simple CRUD: list policy rules, add/edit/delete, view source trust assignments.
- **D-13:** **Primary ingestion validation targets for Phase 4.1:**
  - Official legal PDF documents: Ministry of Transport circulars, Decree 168/2024/NĐ-CP, related government decrees and resolutions
  - Official government portals (HTML): moj.gov.vn, bocongan.gov.vn, Ministry of Transport portal
  - Legal reference websites (thuvienphapluat.vn, luatvietnam.vn) are `SECONDARY` tier — useful as consolidated references but **not primary citation ground truth**. Phase 4.1 does not lock them as authoritative sources.

<specifics>
## Specific Ideas

- **Traceability over coverage:** Explicit structured logging at the four operational boundaries matters more than broad method tracing. The AOP aspect provides dev-time debugging; the explicit logs provide production-grade traceability.
- **Trust policy does NOT replace approval gating:** The trust tier is an additional metadata dimension. All sources still require manual admin approval via the existing `ApprovalState` lifecycle before becoming retrieval-eligible. Trust tier is used for display, filtering, and citation grounding decisions — not to bypass the security gate.
- **Official PDFs first, reference sites second:** When testing batch ingestion against real Vietnamese legal sources, official government legal documents are the primary citation truth. thuvienphapluat.vn and similar aggregators are useful secondary references but should not be the authoritative retrieval baseline.
- **Batch endpoint reuses existing pipeline:** No new orchestration logic — the batch endpoint is a thin coordinator that creates `KbSource` records and triggers existing `IngestionService` methods per item. The existing async job, parser, chunker, and embedder chain handles the rest.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `IngestionService` — existing per-source ingestion trigger; batch endpoint delegates to this per item.
- `SourceAdminController` + `SourceService` — source lifecycle pattern (create, approve, activate) that trust policy complements without replacing.
- `GlobalExceptionHandler` — already uses `ProblemDetail` RFC7807 format; hardening adds cases without changing the response shape.
- `AsyncConfig` — existing executor bean with pool settings read from `application.yaml`; upgrade adds `AsyncConfigurer` without changing pool config.
- Phase 4 `frontend/` — existing admin UI with shadcn DataTable + TanStack Query; trust policy screen follows the same component conventions.

### Established Patterns
- Error responses use `ProblemDetail` (RFC7807) — maintain this in all new exception handler cases.
- `AppException` + `ErrorCode` pattern — new exception types (e.g., `DUPLICATE_BATCH_SOURCE`) should follow this pattern.
- `ApiPaths` constants — all new endpoints get path constants here first.
- Source approval lifecycle (PENDING → APPROVED → ACTIVE) — trust tier is orthogonal metadata, not a replacement or override.
- Domain-oriented naming (01.1 D-17) — no `SpringAi...` prefixes in any new components.

### Integration Points
- New `LoggingAspect` wires in via `@Configuration` + `@Bean` gated by `@Profile("dev")`.
- New trust policy entity connects to `KbSource` via a join or a separate policy table with domain pattern matching.
- Batch import endpoint creates `KbSource` records and triggers existing async jobs — no new executor or queue needed.
- Trust policy admin screen in `frontend/app/(admin)/trust-policy/page.tsx` calling new REST endpoints.

</code_context>

<deferred>
## Deferred Ideas

- **Use-case management** (scenario templates, curated Q&A, document-extracted procedures, intent classifiers) — deferred to a future phase. The Track C configurability audit may identify design signals that inform that future phase, but no use-case implementation happens in Phase 4.1.
- **Dedup, refresh, and scheduling for batch ingestion** — not locked in Phase 4.1. The 409-on-duplicate behavior handles immediate dedup; refresh scheduling and automated re-ingestion are operational concerns for a future phase.
- **Legal reference website sourcing as primary ground truth** — thuvienphapluat.vn and similar are secondary tier in Phase 4.1. Elevating them to primary would require a dedicated curation review in a future phase.
- **Structured JSON logging / Logstash integration** — the JHipster `LoggingConfiguration` Logstash appender is not adopted in Phase 4.1. CRLF prevention is the only log infrastructure change this phase.

</deferred>
