### CR-02: AsyncConfig.getAsyncExecutor() creates a second disconnected executor

**Files modified:** `src/main/java/com/vn/traffic/chatbot/common/config/AsyncConfig.java`
**Commit:** 07ad41e
**Applied fix:** Added an `@Autowired` field `ThreadPoolTaskExecutor ingestionExecutor` and changed `getAsyncExecutor()` to return that injected field instead of creating a new `ThreadPoolTaskExecutor`. Moved the pool configuration (corePoolSize=4, maxPoolSize=10, queueCapacity=100, threadNamePrefix="ingestion-") from `getAsyncExecutor()` into the `@Bean` method `ingestionExecutor()` using `ThreadPoolTaskExecutorBuilder`. Added `org.springframework.beans.factory.annotation.Autowired` import. This ensures a single executor instance is used for both named `@Async("ingestionExecutor")` and default `@Async` invocations, with the uncaught exception handler covering both.

### WR-01: Batch endpoint exposes raw exception messages to API callers

**Files modified:** `src/main/java/com/vn/traffic/chatbot/ingestion/api/IngestionAdminController.java`
**Commit:** db8cc30
**Applied fix:** Split the single `catch (Exception ex)` block into two: a `catch (AppException ex)` block that exposes the known application error message (safe, controlled), and a `catch (Exception ex)` block that logs the full exception but returns a generic `"Ingestion failed"` message to the caller. Added `com.vn.traffic.chatbot.common.error.AppException` import. (This fix was committed together with WR-02 and WR-03.)
