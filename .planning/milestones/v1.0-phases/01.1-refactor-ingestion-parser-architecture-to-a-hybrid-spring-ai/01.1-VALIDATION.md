# Phase 01.1: Validation

**Created:** 2026-04-08
**Phase:** 01.1-refactor-ingestion-parser-architecture-to-a-hybrid-spring-ai
**Purpose:** Define the automated validation gates for the Phase 01.1 hybrid ingestion refactor so provenance, SSRF safety, parser auditability, and metadata compatibility remain intact while the parser internals evolve.

## Validation Scope

This validation artifact covers the checker-required execution gates for:
- `01.1-01-PLAN.md` — URL provenance bugfix and safe fetch/result seam
- `01.1-02-PLAN.md` — Spring AI HTML adapter behind `ParsedDocument`
- `01.1-03-PLAN.md` — bounded Spring AI PDF evaluation with parity gating

It intentionally validates planning promises only. It does not expand scope beyond the approved phase context.

## Validation Principles

1. Preserve the custom trust boundary for URL ingestion.
2. Prove requested URL and resolved final URL persistence with automated tests.
3. Preserve `ParsedDocument` as the anti-corruption layer between Spring AI readers and the rest of ingestion.
4. Preserve chunk metadata required by inspection and retrieval flows.
5. Allow Spring AI PDF adoption only if parity tests prove page-aware provenance compatibility.

## Execution Gates

```bash
./gradlew test --tests "com.vn.traffic.chatbot.ingestion.fetch.SafeUrlFetcherTest" --tests "com.vn.traffic.chatbot.ingestion.orchestrator.IngestionOrchestratorUrlProvenanceTest"
```

Pass conditions:
- Redirect cases persist different requested/final URLs when redirects occur
- Unsafe hosts are blocked before fetch
- Existing ingestion service contract remains green

## Failure Interpretation

| Failure Pattern | Meaning | Required Response |
|----------------|---------|-------------------|
| Requested/final URL tests fail | Provenance bug is not actually fixed end-to-end | Rework fetch seam/orchestrator snapshot flow before proceeding |
| SSRF test fails | Trust boundary leaked or validation ordering regressed | Restore custom pre-fetch validation before any Spring AI work continues |
| HTML adapter tests fail due to Spring AI type leakage | Anti-corruption boundary is broken | Re-map to `ParsedDocument` and keep library types internal |
| Metadata compatibility tests fail | Retrieval/admin contracts drifted | Restore existing metadata keys and chunking behavior |
| PDF parity tests fail | Spring AI PDF path is not provenance-safe yet | Keep Tika as active parser and treat Spring AI PDF as evaluation-only |

## Exit Criteria

Phase 01.1 is validation-complete only when all of the following are true:
- URL provenance defect is proven fixed with automated tests
- SSRF validation remains custom and pre-fetch
- HTML hybrid path preserves `ParsedDocument` and metadata compatibility
- PDF adoption is gated by parity, not assumed
- Full test suite and compile gate pass in a healthy environment
