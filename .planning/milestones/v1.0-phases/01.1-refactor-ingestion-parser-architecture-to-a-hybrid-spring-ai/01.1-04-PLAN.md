<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SafeUrlFetcher -> default HTML reader path | Untrusted fetched content crosses into Spring AI ETL only after custom validation and provenance capture. |
| Reader output -> ParsedDocument | External library output is translated into repo-owned domain records and parser audit fields. |
| TokenChunkingService -> VectorStore / ChunkInspectionService | Token splitting must preserve the metadata contract used for retrieval and admin inspection. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01.1-17 | T | reader-to-domain mapping | mitigate | Keep a repo-owned anti-corruption mapping into `ParsedDocument`; do not let library `Document` objects escape. |
| T-01.1-18 | T | token chunk metadata | mitigate | Assert compatibility for `chunkOrdinal`, `pageNumber`, `sectionRef`, `contentHash`, and retrieval metadata in automated tests. |
| T-01.1-19 | E | trust-boundary erosion | mitigate | Keep SSRF validation, redirect capture, provenance persistence, and job orchestration outside Spring AI readers. |
| T-01.1-20 | R | parser/chunker auditability | mitigate | Persist parser and chunking identity/version so admins can trace how content was ingested and split. |
| T-01.1-21 | D | broad migration regressions | mitigate | Migrate remaining file types and token splitting incrementally behind parity tests and keep narrow fallback paths where parity is not yet proven. |
</threat_model>
